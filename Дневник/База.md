## 1. Переменная

**Переменная** — именованная «коробка» в памяти, где хранится значение. Имя — чтобы обращаться к данным по-человечески, а не по адресу в памяти.

```js
let age = 25;        // объявили и положили значение
age = 26;            // поменяли значение
const PI = 3.14;     // константа — переприсвоить нельзя
```

**`let` / `const` / `var`:**

| Ключевое слово | Переприсваивание | Область видимости | Используют? |
| -------------- | ---------------- | ----------------- | ----------- |
| `const`        | ❌ нельзя         | блок `{}`         | ✅ по умолчанию |
| `let`          | ✅ можно          | блок `{}`         | ✅ когда значение меняется |
| `var`          | ✅ можно          | функция           | ❌ устарел |

> [!warning] Почему не используют `var`
> `var` «всплывает» (hoisting) на верх функции и игнорирует блоки `{}` — отсюда неожиданные баги. Правило джуна: **по умолчанию `const`, если значение меняется — `let`, `var` не пишем**.

**Типы значений** (JS типизирован динамически — тип определяется значением, а не объявлением):
- Примитивы: `string`, `number`, `boolean`, `null`, `undefined`, `symbol`, `bigint`.
- Объекты: `object`, массивы, функции — всё, что сложнее примитива.

> [!note] Примитив vs ссылка
> Примитивы копируются **по значению**, объекты — **по ссылке** (копируется адрес, а не сами данные). Это причина «почему изменил один массив, а поменялись оба». Подробнее — [[JavaScript]].

## 2. Условные операторы и циклы

**Условия** — код выбирает путь в зависимости от проверки.

```js
if (age >= 18) {
  console.log('взрослый');
} else if (age >= 14) {
  console.log('подросток');
} else {
  console.log('ребёнок');
}

// switch — когда сравниваем одну переменную с набором значений
switch (role) {
  case 'admin':  showPanel(); break;
  case 'user':   showFeed();  break;
  default:       showLogin();
}

// тернарник — короткое if/else, возвращает значение
const status = age >= 18 ? 'можно' : 'нельзя';
```

**Логические операторы:** `&&` (И), `||` (ИЛИ), `!` (НЕ), `??` (если `null`/`undefined`).

```js
const name = user.name ?? 'Гость';   // дефолт, если null/undefined
const canEdit = isOwner && isLoggedIn;
```

**Циклы** — повторяют код:

```js
for (let i = 0; i < 5; i++) { ... }          // классический счётчик
for (const item of array) { ... }            // по значениям массива
for (const key in object) { ... }            // по ключам объекта
while (condition) { ... }                    // пока условие истинно
do { ... } while (condition);                // тело выполнится хотя бы раз
```

- `break` — выйти из цикла полностью.
- `continue` — пропустить текущую итерацию, перейти к следующей.

> [!tip] На практике
> Для массивов чаще пишут не циклы, а методы `map` / `filter` / `reduce` — это **декларативно** (что сделать), а не императивно (как обходить). См. [[JavaScript]].

## 3. Отладка кода

**Баг** — ошибка в коде. **Отладка (debugging)** — поиск причины и исправление. Это половина работы программиста.

**Методичный подход** (а не «тыкать наугад»):
1. **Воспроизвести** — добиться, чтобы баг повторялся стабильно.
2. **Локализовать** — сузить место: где именно ломается?
3. **Понять причину** — почему данные/логика не те, что ожидал.
4. **Исправить и проверить** — убедиться, что починилось и ничего не сломалось рядом.

**Инструменты:**

```js
console.log(value);             // самый частый способ — посмотреть значение
console.table(arrayOfObjects);  // данные таблицей
console.error('что-то не так'); // красным, со стеком
debugger;                       // точка останова — код замрёт тут (при открытом DevTools)
```

- **Breakpoints** в DevTools (`F12 → Sources`) — поставить точку останова, выполнять код по шагам, смотреть значения переменных в моменте. Мощнее, чем `console.log`.
- **Stack trace** (стек вызовов) в ошибке — читать **сверху вниз**: верхняя строка = где упало, ниже = кто вызвал.

**Частые ошибки новичка:**

| Ошибка | Что значит |
| ------ | ---------- |
| `ReferenceError: x is not defined` | переменной `x` не существует / опечатка |
| `TypeError: cannot read ... of undefined` | обратился к свойству у `undefined`/`null` |
| `SyntaxError` | сломан синтаксис (скобка, запятая) |
| `NaN` в расчётах | где-то посчитал с не-числом |

> [!tip] Приёмы
> **Линтер** (ESLint) ловит ошибки до запуска. **«Метод утёнка»** — объясни баг вслух (хоть утке) — часто причина становится очевидной в процессе. Гугли **текст ошибки целиком**.

## 4. Алгоритмы и структуры данных

**Структура данных** — способ хранить и организовывать данные. **Алгоритм** — пошаговый рецепт решения задачи. Правильная структура → простой и быстрый алгоритм.

**Базовые структуры:**

| Структура | Идея | Сильна в |
| --------- | ---- | -------- |
| Массив | элементы по индексу подряд | доступ по индексу `O(1)` |
| Связный список | узлы со ссылкой на следующий | вставка/удаление `O(1)` |
| Стек (LIFO) | «стопка тарелок», последним пришёл — первым ушёл | undo, рекурсия |
| Очередь (FIFO) | «очередь в кассу», первым пришёл — первым ушёл | задачи, буферы |
| Хеш-таблица | ключ → значение (объект/`Map` в JS) | поиск по ключу `O(1)` |
| Дерево | иерархия (родитель → дети) | DOM, файлы, индексы БД |
| Граф | узлы и связи | соцсети, маршруты |

**Алгоритмы, которые надо знать:**
- Поиск: линейный `O(n)`, **бинарный** `O(log n)` (только в отсортированном).
- Сортировки: пузырёк (учебный), быстрая (quicksort), слиянием (merge sort).
- Обходы деревьев/графов: в глубину (DFS), в ширину (BFS).

**Big O** — как растёт время/память при росте данных `n` (оценка «в худшем случае»):

| Нотация | Название | Пример |
| ------- | -------- | ------ |
| `O(1)` | константа | доступ к элементу массива |
| `O(log n)` | логарифм | бинарный поиск |
| `O(n)` | линейно | проход по массиву |
| `O(n log n)` | хорошие сортировки | merge / quick sort |
| `O(n²)` | квадрат | вложенные циклы |

> [!note] Подробнее
> Полноценный разбор со схемами и задачами — в [[Алгоритмы и структуры данных]]. Джуну: понять Big O, массив/объект/`Map`/`Set`, стек и очередь, бинарный поиск.

## 5. Операции, функции, классы

**Операции (операторы)** — действия над значениями:

```js
// арифметика
2 + 3, 10 - 4, 6 * 7, 20 / 5, 17 % 5 /* остаток */, 2 ** 10 /* степень */

// сравнение  (=== строгое: без приведения типов)
5 === 5     // true
5 === '5'   // false   ← разные типы
5 == '5'    // true    ← == приводит типы, поэтому его избегают

// присваивание
x += 1; x *= 2;
```

**Функция** — переиспользуемый блок кода: получает вход (аргументы), возвращает результат.

```js
// объявление функции
function sum(a, b) {
  return a + b;
}

// функциональное выражение
const sum2 = function (a, b) { return a + b; };

// стрелочная функция (короткая, без своего this)
const sum3 = (a, b) => a + b;

// параметры по умолчанию
function greet(name = 'Гость') { return `Привет, ${name}`; }
```

> [!tip] Чистые функции
> **Чистая функция** при одинаковом входе всегда даёт одинаковый выход и **не лезет наружу** (не меняет глобальные данные, не пишет в файл/сеть). Такие легко тестировать и переиспользовать. Изменение внешнего мира — **побочный эффект** (side effect).

**Класс** — шаблон для создания объектов с данными и поведением (подробно — раздел [[#7. Классы]]):

```js
class User {
  constructor(name) { this.name = name; }
  hello() { return `Я ${this.name}`; }
}
const u = new User('Аня');
u.hello(); // "Я Аня"
```

## 6. ООП и его принципы

**ООП (объектно-ориентированное программирование)** — стиль, где программа строится из **объектов**, объединяющих **данные** (поля) и **поведение** (методы). Цель — управлять сложностью больших программ.

- **Класс** — чертёж/шаблон (например, «Машина»).
- **Объект (экземпляр)** — конкретная вещь по чертежу («моя красная машина»).

### 4 принципа ООП

**1. Инкапсуляция** — спрятать внутренние данные, дать доступ через методы. Снаружи — «кнопки», а не провода.
```js
class BankAccount {
  #balance = 0;                 // приватное поле (снаружи не достать)
  deposit(sum) { this.#balance += sum; }
  getBalance() { return this.#balance; }
}
```

**2. Наследование** — новый класс берёт поведение родителя и расширяет его. Не дублируем код.
```js
class Animal { eat() { return 'ем'; } }
class Dog extends Animal { bark() { return 'гав'; } }
new Dog().eat(); // "ем" — досталось от Animal
```

**3. Полиморфизм** — «много форм»: один и тот же вызов работает по-разному у разных классов.
```js
class Cat extends Animal { sound() { return 'мяу'; } }
class Cow extends Animal { sound() { return 'му';  } }
[new Cat(), new Cow()].forEach(a => console.log(a.sound())); // мяу, му
```

**4. Абстракция** — выделяем суть, прячем детали. Ты жмёшь «газ», не думая про впрыск топлива. В коде — простой интерфейс над сложной реализацией.

> [!note]
> Альтернатива ООП — **функциональное программирование** (чистые функции, иммутабельность). В современном JS часто смешивают оба подхода.

## 7. Классы

Класс описывает, **из чего** состоит объект (поля) и **что он умеет** (методы).

```js
class User {
  static count = 0;            // статическое поле — общее для всех экземпляров
  #password;                  // приватное поле (только внутри класса)

  constructor(name, password) {
    this.name = name;          // публичное поле экземпляра
    this.#password = password;
    User.count++;
  }

  // метод
  greet() { return `Привет, я ${this.name}`; }

  // геттер — читается как свойство: user.isAdmin
  get isAdmin() { return this.name === 'root'; }

  // статический метод — вызывается у класса, не у экземпляра
  static create(name) { return new User(name, '0000'); }
}

const u = new User('Аня', 'secret');
u.greet();          // "Привет, я Аня"
u.isAdmin;          // false
User.count;         // 1
User.create('Боб'); // фабрика через статический метод
```

Ключевое:
- **`constructor`** — вызывается при `new`, инициализирует поля.
- **`this`** — ссылка на текущий объект.
- **`extends` / `super`** — наследование и вызов конструктора/методов родителя.
- **`#поле`** — приватное (инкапсуляция).
- **`static`** — принадлежит классу, а не экземпляру.

```js
class Admin extends User {
  constructor(name) {
    super(name, 'root-pass'); // обязателен вызов super() до использования this
    this.role = 'admin';
  }
}
```

> [!note] Под капотом JS
> В JS классы — «синтаксический сахар» над **прототипами**: методы лежат в прототипе, а объекты ссылаются на него. Поэтому метод существует в памяти один раз на все экземпляры. Детали — [[JavaScript]].

## 8. SOLID

**SOLID** — 5 принципов хорошего ООП-дизайна. Цель — код, который легко менять и тестировать.

| Буква | Принцип | Суть «на пальцах» |
| ----- | ------- | ----------------- |
| **S** | Single Responsibility | у класса/модуля **одна** причина для изменения (одна задача) |
| **O** | Open/Closed | открыт для **расширения**, закрыт для **изменения** (добавляй, не переписывай) |
| **L** | Liskov Substitution | наследник должен **подменять** родителя, не ломая программу |
| **I** | Interface Segregation | много **узких** интерфейсов лучше одного «толстого» |
| **D** | Dependency Inversion | зависеть от **абстракций**, а не от конкретных классов |

**Пример S (Single Responsibility) — плохо и хорошо:**
```js
// ❌ класс делает всё сразу: и данные, и сохранение, и письмо
class User {
  save() { /* пишет в БД */ }
  sendEmail() { /* шлёт письмо */ }
}

// ✅ разделили ответственность
class User { /* только данные пользователя */ }
class UserRepository { save(user) { /* БД */ } }
class EmailService  { send(user)  { /* письмо */ } }
```

**Пример D (Dependency Inversion):**
```js
// ❌ жёстко завязан на MySQL
class Service { db = new MySQL(); }

// ✅ принимает любое хранилище через абстракцию
class Service {
  constructor(storage) { this.storage = storage; } // MySQL, Postgres, мок для теста
}
```

> [!note]
> Джуну достаточно **понимать S и D** и узнавать остальные. Подробнее с примерами — [[Архитектура и паттерны]].

## 9. Паттерны проектирования

**Паттерн** — типовое, проверенное решение часто встречающейся задачи проектирования. Это **не код для копирования**, а идея/шаблон. Три категории:

**Порождающие** (как создавать объекты):
- **Singleton** — один-единственный экземпляр на всё приложение (конфиг, логгер).
- **Factory** — отдельный метод/класс решает, какой объект создать.

**Структурные** (как собирать объекты в большие структуры):
- **Adapter** — переходник между несовместимыми интерфейсами.
- **Facade** — простой фасад над сложной подсистемой.

**Поведенческие** (как объекты общаются):
- **Observer** — объекты подписываются на события и реагируют (на нём построен UI и события в JS).
- **Strategy** — взаимозаменяемые алгоритмы, выбираемые на лету.

```js
// Singleton
class Config {
  static #instance;
  static get() { return (Config.#instance ??= new Config()); }
}
Config.get() === Config.get(); // true — всегда один объект

// Observer (наблюдатель)
class Emitter {
  #subs = [];
  subscribe(fn) { this.#subs.push(fn); }
  emit(data)    { this.#subs.forEach(fn => fn(data)); }
}
```

> [!warning] Не переусердствуй
> Паттерны решают **реальную** сложность. Тащить их в простой код = усложнять на ровном месте. Сначала простое решение — паттерн, когда заболело. Подробнее — [[Архитектура и паттерны]].

## 10. Асинхронность, многопоточность, параллелизм

Три похожих, но разных понятия:

- **Конкурентность (concurrency)** — несколько задач **в работе одновременно**, но переключаемся между ними (один повар жонглирует несколькими блюдами).
- **Параллелизм (parallelism)** — задачи реально выполняются **в один момент** на разных ядрах (несколько поваров).
- **Асинхронность** — не ждём долгую операцию (сеть, диск) блокируя всё; запустили и продолжили, результат придёт позже.

**JavaScript однопоточный** — один «поток» исполняет код. Чтобы не зависать на долгих операциях, есть **Event Loop**: долгие задачи (таймеры, запросы) уходят в браузер/ОС, а их результат возвращается в очередь и обрабатывается, когда поток свободен.

**Эволюция асинхронного кода:**
```js
// 1) callbacks — легко получить «ад колбэков»
getUser(id, (user) => getOrders(user, (orders) => render(orders)));

// 2) Promise — цепочки .then
getUser(id).then(getOrders).then(render).catch(handleError);

// 3) async/await — выглядит как синхронный, читается сверху вниз
async function show(id) {
  try {
    const user   = await getUser(id);
    const orders = await getOrders(user);
    render(orders);
  } catch (e) { handleError(e); }
}
```

**Многопоточность / параллелизм** реально дают:
- В браузере — **Web Workers** (тяжёлые вычисления в отдельном потоке, не блокируя UI).
- В Node.js — **Worker Threads** и кластеризация процессов.

**Процесс vs поток:** процесс — отдельная программа со своей памятью; поток — «ветка» внутри процесса, потоки делят память (быстрее общаются, но риск гонок данных).

> [!note]
> На собесе джуна почти всегда про **event loop, микро/макрозадачи и `Promise`**, а не про настоящие потоки. Разбор event loop — [[JavaScript]].

## 11. Что такое сайт и сервер

- **Сайт** — набор связанных страниц и файлов (HTML, CSS, JS, картинки), доступных по адресу в интернете.
- **Сервер** — два значения:
  - *Железо* — постоянно включённый компьютер в дата-центре.
  - *Программа* — софт (nginx, Apache, Node.js), который принимает запросы и отдаёт ответы.

**Клиент-серверная модель:** диалог всегда начинает **клиент** (браузер): «дай страницу» → **сервер**: «держи». Сервер сам по себе ничего не присылает (исключение — WebSocket, где соединение держится открытым).

```
Клиент (браузер)  ⇄  Сервер (отдаёт HTML/CSS/JS, данные из БД)
```

> [!note] Подробно
> Протоколы (HTTP, AJAX, WebSocket) и схемы — в [[Сеть и HTTP]] и дневнике за 23.06.

## 12. Что происходит, когда открываешь сайт

Классический вопрос собеса. Скелет: **URL → DNS → TCP → (TLS) → HTTP-запрос → ответ → рендер**.

1. **Парсинг URL** — браузер разбирает адрес на части (протокол, домен, путь…).
2. **DNS** — переводит домен в IP-адрес (сначала проверяется кэш).
3. **TCP-соединение** — браузер устанавливает связь с сервером (рукопожатие).
4. **TLS-рукопожатие** — для `https` настраивается шифрование.
5. **HTTP-запрос** — браузер шлёт `GET /path`.
6. **Сервер обрабатывает** и возвращает HTTP-ответ (HTML + статус-код).
7. **Рендеринг** — браузер строит **DOM**-дерево, грузит CSS/JS/картинки (доп. запросы), считает стили, раскладывает (layout) и рисует (paint) страницу.

> [!tip]
> Хорошо ещё упомянуть: после загрузки HTML браузер находит ссылки на CSS/JS/картинки и делает **дополнительные запросы** за ними. JS может догружать данные через `fetch` уже после отрисовки.

## 13. Что такое фронтенд и бэкенд

| | **Фронтенд** | **Бэкенд** |
| --- | --- | --- |
| Где работает | в браузере (клиент) | на сервере |
| Отвечает за | то, что видит и трогает юзер | логику, данные, безопасность |
| Технологии | HTML, CSS, JS, React/Vue | Node.js, Python, Go, БД |
| Видно пользователю? | да (интерфейс) | нет (скрыто) |

- **Фронтенд** — интерфейс: вёрстка, стили, поведение кнопок, отправка запросов на бэк.
- **Бэкенд** — мозги: проверяет права, считает, ходит в базу данных, отдаёт фронту данные (обычно JSON через API).
- **Fullstack** — и то, и другое.

```
[Фронтенд] --запрос(HTTP)--> [Бэкенд] --SQL--> [База данных]
[Фронтенд] <--JSON-----------  [Бэкенд] <-------
```

> [!note]
> Правило безопасности: **никогда не доверяй фронтенду**. Проверки (цена, права, валидация) дублируются на бэке — клиентский код можно подменить. См. [[Аутентификация и безопасность]].

## 14. Что такое HTTP-запросы

**HTTP** — протокол, по которому клиент и сервер обмениваются сообщениями. Запрос = **метод + URL + заголовки + (тело)**.

**Методы** (что сделать с ресурсом):

| Метод | Зачем | Идемпотентен* |
| ----- | ----- | ------------- |
| `GET` | получить данные | да |
| `POST` | создать новое | нет |
| `PUT` | заменить целиком | да |
| `PATCH` | обновить часть полей | нет |
| `DELETE` | удалить | да |

\* *Идемпотентность* — повторный одинаковый запрос даёт тот же результат (`GET` дёргай хоть 100 раз; `POST` дважды → две записи).

**Статус-коды** (первая цифра = категория):
- **2xx успех:** `200 OK`, `201 Created`, `204 No Content`.
- **3xx редирект:** `301` (навсегда), `302` (временно), `304` (бери из кэша).
- **4xx ошибка клиента:** `400` (кривой запрос), `401` (не вошёл), `403` (вошёл, но нет прав), `404` (нет ресурса).
- **5xx ошибка сервера:** `500` (упал), `502/503` (шлюз/сервис недоступен).

```http
POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json     ← формат тела
Authorization: Bearer eyJ...       ← токен

{ "name": "Аня" }                  ← тело
```

**HTTP vs HTTPS:** HTTPS = HTTP + TLS-шифрование. Сегодня всё должно быть на HTTPS.

> [!note] Подробно
> Заголовки, CORS, `fetch`, DevTools Network — [[Сеть и HTTP]] и дневник за 23.06.

## 15. Что такое DNS и как оно работает

**DNS (Domain Name System)** — «телефонная книга интернета»: переводит человекочитаемый домен (`google.com`) в IP-адрес (`142.250.150.139`), по которому реально находят сервер.

**Как идёт резолв** (по цепочке, пока не найдём; на каждом шаге работает кэш):
1. **Кэш браузера** → **кэш ОС** → файл `hosts`.
2. **Рекурсивный резолвер** (обычно у провайдера или `8.8.8.8` Google).
3. **Корневой сервер** (`.`) → подскажет, кто отвечает за зону `.com`.
4. **TLD-сервер** (`.com`) → подскажет авторитетный сервер домена.
5. **Авторитетный сервер** `google.com` → отдаёт финальный IP.

```
google.com → [кэш?] → резолвер → корневой(.) → TLD(.com) → авторитетный → IP
```

**Типы записей:**

| Запись | Назначение |
| ------ | ---------- |
| `A` | домен → IPv4 |
| `AAAA` | домен → IPv6 |
| `CNAME` | псевдоним на другой домен |
| `MX` | почтовый сервер |
| `TXT` | произвольный текст (верификация, SPF) |
| `NS` | какие сервера обслуживают зону |

> [!tip]
> Из-за кэширования изменения DNS «расходятся» не мгновенно (зависит от **TTL** записи). Поэтому новый сайт «у кого-то уже открылся, у кого-то ещё нет».

## 16. Что такое базы данных и их виды

**База данных (БД)** — организованное хранилище данных. **СУБД** — программа, которая этим хранилищем управляет (PostgreSQL, MySQL, MongoDB, Redis).

**Два больших мира:**

**1. Реляционные (SQL)** — данные в **таблицах** со строками и столбцами, связи между таблицами, строгая схема. Язык — **SQL**.
- Примеры: PostgreSQL, MySQL, SQLite.
- Сильны в: целостности данных, сложных связях, транзакциях.
```sql
SELECT name, email FROM users WHERE age > 18 ORDER BY name;
```

**2. Нереляционные (NoSQL)** — гибкая или отсутствующая схема. Виды:

| Тип | Идея | Пример |
| --- | ---- | ------ |
| Документные | JSON-документы | MongoDB |
| Ключ-значение | словарь key→value, очень быстро | Redis |
| Колоночные | для аналитики на больших данных | Cassandra |
| Графовые | узлы и связи | Neo4j |

**ACID** (гарантии надёжных транзакций в SQL): **A**tomicity (всё или ничего), **C**onsistency (данные остаются валидными), **I**solation (транзакции не мешают друг другу), **D**urability (сохранилось — не пропадёт).

> [!note] Что выбрать
> Джуну по умолчанию — **реляционная БД (PostgreSQL)**: предсказуемо и универсально. NoSQL берут под конкретную задачу (кэш → Redis, гибкие документы → Mongo). Подробно с SQL — [[Базы данных и SQL]].

## 17. Что такое кэш и кэширование

**Кэш** — временное хранилище для быстрого повторного доступа к данным. **Кэширование** — приём «не считать/не запрашивать то, что уже получали».

Зачем: **скорость** (близко и готово) + **разгрузка** (меньше дёргаем сервер/БД).

**Уровни кэша в вебе:**
- **Браузерный кэш** — картинки, CSS, JS не качаются повторно (управляется заголовками `Cache-Control`, `ETag`).
- **CDN** — копии статики на серверах по всему миру, ближе к пользователю.
- **Серверный кэш / Redis** — часто запрашиваемые данные держим в памяти, не ходим каждый раз в БД.
- **Кэш БД** — сама СУБД кэширует частые запросы.

```js
// простейший кэш в памяти
const cache = new Map();
function getUser(id) {
  if (cache.has(id)) return cache.get(id);   // попадание (hit)
  const user = db.findUser(id);              // промах (miss) → читаем из БД
  cache.set(id, user);
  return user;
}
```

> [!warning] Главная боль — инвалидация
> Шутка индустрии: *«В программировании две сложные вещи: инвалидация кэша и придумывание имён»*. Проблема — **вовремя обновить/сбросить** кэш, чтобы пользователь не видел устаревшие данные. Решения: TTL (время жизни), сброс по событию изменения.

## 18. Что такое библиотека и фреймворк

И то, и другое — **готовый чужой код**, чтобы не писать всё с нуля. Разница — **кто кем управляет**:

- **Библиотека** — набор функций, **ты** вызываешь её, когда нужно. Ты главный. (lodash, axios, day.js)
- **Фреймворк** — каркас, который **вызывает твой код** в нужные моменты (**инверсия управления**). Главный он, ты заполняешь «дырки». (Angular, Next.js, NestJS)

> [!tip] Метафора Голливуда
> «Не звоните нам — мы сами вам позвоним». **Фреймворк звонит тебе**, библиотеку звонишь ты.

```js
// библиотека: я решаю, когда вызвать
import _ from 'lodash';
const unique = _.uniq([1, 1, 2]);   // [1, 2]

// фреймворк: он вызовет мою функцию-компонент сам, когда надо отрисовать
function App() { return <h1>Привет</h1>; }
```

> [!note]
> React формально — библиотека (UI), но с экосистемой (роутер, сборка) ощущается как фреймворк. См. [[React и экосистема]]. Граница размыта — важно понимать **кто кого вызывает**.

## 19. Что такое CRUD

**CRUD** — 4 базовые операции над данными, есть в любом приложении с хранилищем:

| CRUD | Действие | HTTP | SQL |
| ---- | -------- | ---- | --- |
| **C**reate | создать | `POST` | `INSERT` |
| **R**ead | прочитать | `GET` | `SELECT` |
| **U**pdate | изменить | `PUT`/`PATCH` | `UPDATE` |
| **D**elete | удалить | `DELETE` | `DELETE` |

Пример для «пользователей»:
```
POST   /users      → создать
GET    /users/42   → прочитать
PATCH  /users/42   → обновить
DELETE /users/42   → удалить
```

> [!note]
> Почти любое приложение (блог, магазин, todo) — это CRUD над какими-то сущностями + бизнес-логика сверху.

## 20. Что такое API

**API (Application Programming Interface)** — «контракт»/интерфейс, через который одна программа общается с другой, **не зная её внутренностей**. Розетка — это API электросети: воткнул вилку, не вникая в устройство станции.

В вебе чаще всего: **фронт ↔ бэк** через HTTP, данные в **JSON**.

**Виды веб-API:**

| Стиль | Кратко |
| ----- | ------ |
| **REST** | ресурсы в URL + HTTP-методы; самый распространённый |
| **GraphQL** | один endpoint, клиент сам описывает, какие поля хочет |
| **gRPC** | бинарный, быстрый, для общения сервисов между собой |
| **SOAP** | старый, XML, в энтерпрайзе/легаси |
| **WebSocket** | двусторонний реалтайм (чаты, игры) |

```js
// дёргаем чужой API
const res = await fetch('https://api.github.com/users/torvalds');
const user = await res.json();
console.log(user.name); // "Linus Torvalds"
```

> [!note] Подробно
> Принципы REST, проектирование endpoint’ов, коды и доку — [[REST API на сервере]] и [[Сеть и HTTP]].

## 21. Что такое Deploy, DevOps и CI-CD

- **Deploy (деплой)** — «выкатка»: доставить код на сервер, где им пользуются (продакшен). Из «работает у меня» → «работает у всех».
- **DevOps** — культура и практики, сближающие **разработку (Dev)** и **эксплуатацию (Ops)**: автоматизация сборки, тестов, доставки, мониторинга. Меньше ручного, меньше «у меня же работало».
- **CI/CD** — конвейер автоматизации:
  - **CI (Continuous Integration)** — на каждый пуш автоматически собрать и **прогнать тесты/линтер**. Цель — рано ловить поломки.
  - **CD (Continuous Delivery/Deployment)** — автоматически **доставить/выкатить** прошедшую проверки версию.

**Типичный пайплайн:**
```
push в git → [CI: установка → линт → тесты → сборка] → [CD: деплой на сервер]
                         ❌ упало — выкатки нет
```

> [!tip]
> Польза для джуна: видно сразу, что твой PR не ломает сборку/тесты. Подробнее про серверную часть и инфраструктуру — [[Серверная архитектура и DevOps]] и [[Git и инструменты]].

## 22. Что такое gitlab-ci файл и как его настроить

**`.gitlab-ci.yml`** — файл в корне репозитория, описывающий пайплайн CI/CD для **GitLab**. GitLab при каждом пуше читает его и запускает работы на **раннере** (исполнителе).

**Понятия:**
- **stage** — этап (например `test`, `build`, `deploy`); этапы идут по порядку.
- **job** — конкретная работа внутри этапа (команды в `script`).
- **runner** — машина/контейнер, где выполняются джобы.

**Пример для фронтенд-проекта:**
```yaml
stages:        # порядок этапов
  - install
  - test
  - build

default:
  image: node:20          # в каком окружении запускать

install_deps:
  stage: install
  script:
    - npm ci
  artifacts:              # передать node_modules дальше
    paths: [node_modules/]

lint_and_test:
  stage: test
  script:
    - npm run lint
    - npm test

build_app:
  stage: build
  script:
    - npm run build
  artifacts:
    paths: [dist/]        # сохранить результат сборки
  only:
    - main                # собирать только из ветки main
```

**Как настроить (с нуля):**
1. Положить `.gitlab-ci.yml` в корень репо.
2. Описать `stages` и `job`’ы со `script`.
3. Запушить — GitLab сам запустит пайплайн (вкладка **CI/CD → Pipelines**).
4. Секреты (токены, пароли) — **не в файл**, а в **CI/CD Variables** настроек проекта.

> [!warning]
> Никогда не коммить пароли/токены в `.gitlab-ci.yml`. Только переменные окружения из защищённых настроек.

## 23. Что такое облачная инфраструктура

**Облако (cloud)** — аренда чужих серверов и сервисов через интернет вместо своего «железа» в офисе. Платишь за использование, масштабируешься по кнопке. Провайдеры: **AWS, Google Cloud (GCP), Azure**, из ближних — Yandex Cloud, VK Cloud.

**Модели обслуживания** (сколько отдаёшь провайдеру):

| Модель | Что даёт провайдер | Аналогия |
| ------ | ------------------ | -------- |
| **IaaS** | голые серверы, сеть, диски (ты ставишь ОС и софт) | арендуешь землю |
| **PaaS** | готовая платформа (выкатил код — оно работает) | арендуешь квартиру |
| **SaaS** | готовое приложение (просто пользуешься) | живёшь в отеле |

Примеры: IaaS — AWS EC2; PaaS — Heroku, Vercel; SaaS — Gmail, Notion.

**Связанные технологии:**
- **Контейнеры / Docker** — упаковать приложение со всем окружением в «коробку», которая одинаково запускается где угодно («у меня работало» больше не отговорка).
- **Kubernetes (K8s)** — «дирижёр» контейнеров: сам поднимает, масштабирует, перезапускает упавшие.
- **Serverless** — пишешь только функцию, серверы и масштабирование — забота облака (AWS Lambda).

```
Твой код → Docker-образ → облако (масштабируется под нагрузку) → пользователи
```

> [!note] Для джуна
> Достаточно понимать: **зачем облако** (не держать своё железо, платить за использование, масштабироваться), **IaaS/PaaS/SaaS** и **что такое Docker**. Глубже — [[Серверная архитектура и DevOps]].
