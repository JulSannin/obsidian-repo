## Что я прочитал и освоил

### 1. Клиент ↔ сервер

**Интернет работает по модели клиент-сервер**
	клиент - тот, кто запрашивает(браузер)
	сервер - компьютер, который постояно включен и отдает данные по запросу клиента(html,css,js,файлы)

**Диалог всегда начинает клиент**
	клиент - дай страницу
	сервер - держи страницу

- **Браузер** — программа-клиент: отправляет запросы, получает HTML/CSS/JS и **рисует** страницу.
- **Веб-сервер** — программа на сервере (nginx, Apache, Node.js), которая принимает запросы и отдаёт ответы.

Сервер как правило без запроса ничего не присылает, но есть исключение, это WebSockets

#### Протоколы, по которым может идти диалог клиента и сервера

**HTTP** - Протокол для передачи гипертекстовых данных (Hyper Text Transfer Protocol), который используется повсеместно. HTTP используется в клиент-серверной архитектуре
![[HTTP.jpeg]]
Особенности HTTP:
- HTTP не поддерживает соединение, после того, как отдает ответ на запрос.
- HTTP обязует клиентов заранее оговаривать действие, которое клиент хочет сделать в заголовке (HTTP Headers) - GET, POST, PUT, DELETE .

**AJAX** - асинхронные запросы с помощью JavaScript (Asynchonous JavaScript and XML). AJAX преследует все те же цели, что и HTTP, только делает это уже асинхронно. Если ранее нужно было для каждого запроса прописывать свой URL и перезагружать страницу, то теперь можно просто использовать AJAX и он сам будет отправлять нужные URL серверу и получать данные.
![[AJAX.jpeg]]
Особенности AJAX:
- Все ещё обычный запрос, который не поддерживает соединение, после того, как отдает ответ на запрос.
- Все ещё заранее оговариваем действие, которое клиент хочет сделать в заголовке (HTTP Headers) - GET, POST, PUT, DELETE .
- Теперь мы делаем это асинхронно благодаря JavaScript .

**WebSocket** - протокол для общения между клиентом и сервером, предоставляющий двустороннee общение поверх протокола TCP.
Мы подключаем **WS** один раз, а затем сервер может отдавать нам ответы тогда, когда посчитает нужным:
![[WebSockets.jpeg]]

### 2. URL, домен, IP, DNS

**IP-адрес** — числовой адрес компьютера в сети, например `142.250.150.139`. По нему реально находят сервер. Людям такое запоминать неудобно → придумали домены.

**Домен** — читаемое имя: `google.com`.

**DNS (Domain Name System)** — «телефонная книга интернета». Переводит домен → IP. Ты пишешь `google.com`, DNS говорит «это `142.250.150.139`».

**URL** — полный адрес ресурса. Разбор по частям:
![[URL.png]]
- **схема (protocol)** — `https` / `http`
- **хост (домен)** — какой сервер
- **порт** — дверь на сервере (`80` для http, `443` для https — обычно скрыты)
- **путь (path)** — какой ресурс на сервере
- **query** — параметры после `?`, пары `ключ=значение` через `&`
- **fragment** — после `#`, прыжок к якорю _внутри_ страницы (на сервер не отправляется)

### 3. Что происходит при вводе адреса в браузере

1. **Парсинг URL** — браузер разбирает адрес на части.
2. **DNS-запрос** — домен → IP (сначала проверяется кэш браузера/ОС, потом DNS-сервер).
3. **TCP-соединение** — браузер устанавливает связь с сервером по IP (рукопожатие TCP).
4. **TLS-рукопожатие** — если `https`, шифрование настраивается здесь.
5. **HTTP-запрос** — браузер шлёт `GET /path`.
6. **Сервер обрабатывает** и возвращает **HTTP-ответ** (HTML + статус-код).
7. **Рендеринг** — браузер парсит HTML, строит DOM-дерево, грузит CSS/JS/картинки (доп. запросы), рисует страницу.

Запомнить скелет: **URL → DNS → TCP → (TLS) → HTTP-запрос → ответ → рендер**.

### 4. HTTP vs HTTPS

- **HTTP** — протокол передачи данных. Текст летит **открытым** — кто угодно по пути (Wi-Fi, провайдер) может прочитать/подменить.
- **HTTPS** = HTTP + **TLS-шифрование**. Данные зашифрованы, плюс проверяется, что сервер — настоящий (по сертификату).

Сегодня **всё должно быть на HTTPS**. Без него браузеры ставят «Не защищено», а многие фичи (геолокация, камера, Service Workers) просто не работают.

### 5. HTTP-методы

Метод говорит серверу, **что** ты хочешь сделать с ресурсом:

| Метод    | Зачем                       | Идемпотентность* |
| -------- | --------------------------- | ---------------- |
| `GET`    | получить данные             | да               |
| `POST`   | создать новое               | нет              |
| `PUT`    | заменить ресурс **целиком** | да               |
| `PATCH`  | обновить **часть** полей    | нет              |
| `DELETE` | удалить                     | да               |

_Идемпотентность_ — повторный одинаковый запрос даёт тот же результат. `GET` дёргай хоть 100 раз — данные не меняются. А `POST` дважды → создашь две записи.

**PUT vs PATCH** (частый вопрос): PUT шлёт весь объект и заменяет его целиком; PATCH шлёт только изменённые поля.

### 6. Статус-коды

Сервер в ответе всегда возвращает код. Первая цифра = категория:

- **2xx — успех**
    - `200 OK` — всё ок
    - `201 Created` — создан ресурс (ответ на POST)
    - `204 No Content` — ок, но тела нет
- **3xx — перенаправление**
    - `301 Moved Permanently` — навсегда переехал
    - `302 Found` — временный редирект
    - `304 Not Modified` — бери из кэша
- **4xx — ошибка клиента (ты накосячил)**
    - `400 Bad Request` — кривой запрос
    - `401 Unauthorized` — **не аутентифицирован** (кто ты? войди)
    - `403 Forbidden` — **аутентифицирован, но нельзя** (знаю кто ты, доступа нет)
    - `404 Not Found` — нет такого ресурса
- **5xx — ошибка сервера (сервер накосячил)**
    - `500 Internal Server Error` — упал сервер
    - `502 / 503` — шлюз/сервис недоступен

**401 vs 403** (вопрос-ловушка): 401 — «ты вообще не вошёл», 403 — «вошёл, но прав нет».

### 7. Заголовки, тело, Content-Type

Любой запрос/ответ = **заголовки** (метаданные) + опционально **тело** (данные).

Пример запроса:

```http
POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json      ← какой формат у тела
Authorization: Bearer eyJhbG...     ← токен авторизации
Accept: application/json            ← какой формат хочу в ответ

{ "name": "Аня", "age": 25 }        ← тело
```

- **`Content-Type`** — в каком формате тело: `application/json`, `text/html`, `multipart/form-data` (файлы).
- **`Authorization`** — токен/ключ доступа.
- У `GET` тела обычно нет — параметры идут в query-строке URL.

### 8. REST API — принципы

**API** — интерфейс, через который программы общаются. **REST** — самый распространённый стиль HTTP-API. Идея: **ресурсы** (существительные) в URL + **методы** (глаголы) для действий над ними.

```
GET    /users         → список пользователей
GET    /users/42      → пользователь №42
POST   /users         → создать пользователя
PUT    /users/42      → заменить пользователя 42
PATCH  /users/42      → обновить часть полей
DELETE /users/42      → удалить
```

Заметь: URL — это **что** (ресурс), метод — это **действие**. Плохо: `/getUser?id=42`. Хорошо: `GET /users/42`.

**Как читать доку API:** ищешь — какой `метод` + `endpoint`, какие параметры/тело нужны, что вернётся (формат + коды), нужна ли авторизация. Тренируйся на доках, например, любого публичного API (GitHub, погода).

### 9. fetch и JSON → [JavaScript](app://obsidian.md/JavaScript)

**JSON** — текстовый формат обмена данными, похож на объект JS:

```json
{ "name": "Аня", "skills": ["html", "css"], "active": true }
```

`fetch` — встроенная в браузер функция для HTTP-запросов. Возвращает `Promise`.

```js
// GET
const res = await fetch('https://api.example.com/users/42');
if (!res.ok) throw new Error(`HTTP ${res.status}`); // res.ok = true для 2xx
const user = await res.json();   // парсим тело из JSON в объект JS
console.log(user.name);

// POST
const res = await fetch('https://api.example.com/users', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'Аня' }),  // объект → JSON-строку
});
```

Важно: `fetch` **не падает** на статусах 4xx/5xx — проверяй `res.ok` сам. `res.json()` — тоже Promise (нужен `await`).

### 10. Вкладка Network в DevTools

Открой `F12 → Network`, перезагрузи страницу. Здесь видно **каждый** запрос:

- **Status** — код ответа (зелёный/красный)
- **Method** — GET/POST...
- **Type** — что грузилось (document, fetch/xhr, img, css)
- Клик по запросу → **Headers** (заголовки), **Payload** (что отправили), **Response** (что вернулось), **Timing** (сколько шло).

Это **главный инструмент** для отладки общения с сервером. Фильтр `Fetch/XHR` показывает только API-запросы.

### 11. CORS

**CORS (Cross-Origin Resource Sharing)** — механизм безопасности браузера. По умолчанию JS со страницы `site-a.com` **не может** читать ответы запросов на `api-site-b.com` (другой origin = схема+домен+порт).

- Ошибку CORS ты увидишь в Console: _«blocked by CORS policy»_.
- **Кто чинит:** НЕ фронтенд. Чинит **сервер** (бэкенд) — он должен прислать заголовок `Access-Control-Allow-Origin` с разрешённым origin. На фронте ты CORS «починить» не можешь, только попросить бэкенд или поставить прокси для разработки.
- Это **не** проблема твоего кода — это сервер не разрешил доступ.
