## Что я прочитал и освоил

### 1. Зачем нужен Docker

**Проблема:** «у меня на компе работает, а на сервере — нет». Разные версии Node, библиотек, ОС → приложение ведёт себя по-разному.

**Docker** упаковывает приложение **вместе со всем окружением** (код, рантайм, библиотеки, системные зависимости) в изолированную коробку — **контейнер**. Эта коробка одинаково запускается где угодно: на твоём ноуте, у коллеги, на сервере.

Девиз: **«build once, run anywhere»** — собрал один раз, запускай везде.

### 2. Контейнер vs виртуальная машина

|            | Виртуальная машина (VM)            | Контейнер                       |
| ---------- | ---------------------------------- | ------------------------------- |
| Что внутри | целая гостевая **ОС** + приложение | только приложение + зависимости |
| Ядро       | своё                               | **общее** с хостом              |
| Вес        | гигабайты                          | мегабайты                       |
| Старт      | минуты                             | секунды                         |

VM виртуализирует **железо** и тащит за собой полноценную ОС. Контейнер виртуализирует **ОС**: все контейнеры делят **ядро хоста**, поэтому они лёгкие и стартуют за секунды.

### 3. Главные понятия

- **Образ (image)** — неизменяемый **шаблон**: снимок файловой системы с приложением и зависимостями. Это как **класс** в ООП или установочный диск.
- **Контейнер (container)** — **запущенный экземпляр** образа. Как **объект** класса. Из одного образа можно поднять сколько угодно контейнеров.
- **Dockerfile** — текстовый **рецепт**, по которому собирается образ.
- **Реестр (registry)** — хранилище образов. Публичный — **Docker Hub** (как npm, только для образов). `docker pull node` тянет образ оттуда.

Цепочка: **Dockerfile → (build) → образ → (run) → контейнер**.

### 4. Dockerfile

Рецепт сборки образа. Каждая инструкция = **слой (layer)**, слои кэшируются.

```dockerfile
FROM node:20-alpine        # базовый образ (с чего начинаем)
WORKDIR /app               # рабочая папка внутри контейнера
COPY package*.json ./      # сначала только манифесты...
RUN npm ci                 # ...ставим зависимости (этот слой закэшируется)
COPY . .                   # потом копируем весь код
EXPOSE 3000                # документируем порт (сам по себе не публикует)
CMD ["node", "server.js"]  # команда запуска контейнера
```

**Почему `package.json` копируем отдельно?** Docker кэширует слои. Пока `package.json` не менялся — слой с `npm ci` берётся из кэша, и пересборка летает. Поменял только код → зависимости заново не ставятся.

- **`RUN`** — выполняется при **сборке** образа.
- **`CMD`** — выполняется при **старте** контейнера (в образе он один).

### 5. Основные команды

```bash
docker build -t myapp .          # собрать образ из Dockerfile (тег myapp)
docker images                    # список образов
docker run -p 3000:3000 myapp    # запустить контейнер (хост:контейнер)
docker ps                        # запущенные контейнеры
docker ps -a                     # все, включая остановленные
docker stop <id>                 # остановить
docker rm <id>                   # удалить контейнер
docker rmi myapp                 # удалить образ
docker logs <id>                 # логи контейнера
docker exec -it <id> sh          # зайти внутрь контейнера (шелл)
```

Полезные флаги `run`: `-p хост:контейнер` (порты), `-d` (фоном, detached), `-it` (интерактивно), `--name` (имя), `-e KEY=val` (переменная окружения), `-v` (том).

### 6. Порты и тома

**Проброс портов (`-p 3000:3000`)** — контейнер изолирован, его внутренний порт надо «вывести наружу». Слева — порт на твоём компе, справа — порт внутри контейнера.

**Тома (volumes, `-v`)** — контейнер **эфемерный**: удалил → данные внутри пропали. Чтобы данные жили (БД, загрузки), их кладут в **том** — хранилище вне контейнера.

```bash
docker run -v mydata:/var/lib/postgresql/data postgres  # именованный том
docker run -v $(pwd):/app node                          # папка хоста внутрь (удобно в разработке)
```

### 7. Зачем Docker Compose

Реальное приложение — это не один контейнер: фронт + бэк + БД + кэш. Поднимать каждый руками длинной командой `docker run ...` — боль.

**Docker Compose** описывает **все сервисы в одном YAML-файле** и поднимает их одной командой. Это **дирижёр** для нескольких контейнеров.

### 8. docker-compose.yml

```yaml
services:
  app:                          # сервис 1 — наше приложение
    build: .                    # собрать из локального Dockerfile
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/mydb
    depends_on:
      - db                      # стартовать после db

  db:                           # сервис 2 — база
    image: postgres:16          # готовый образ с Docker Hub
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data   # данные переживут перезапуск

volumes:
  pgdata:                       # объявляем именованный том
```

**Магия сети:** Compose сам создаёт общую сеть, и сервисы видят друг друга **по имени сервиса**. В `DATABASE_URL` хост — это `db` (имя сервиса), а **не** `localhost`. Частая ошибка новичков: внутри Compose обращайся к соседу по имени сервиса, `localhost` укажет на сам контейнер.

### 9. Команды Compose

```bash
docker compose up            # поднять всё (по docker-compose.yml)
docker compose up -d         # ...в фоне
docker compose up --build    # пересобрать образы и поднять
docker compose ps            # статус сервисов
docker compose logs -f app   # логи сервиса app, follow
docker compose exec app sh   # шелл внутри сервиса app
docker compose down          # остановить и удалить контейнеры + сеть
docker compose down -v       # ...вместе с томами (снесёт данные БД!)
```

`build` vs `image` в Compose: `build: .` — собрать свой образ из Dockerfile; `image: postgres:16` — взять готовый из реестра.

### 10. Что запомнить

- **Образ — шаблон, контейнер — запущенный экземпляр** (класс vs объект).
- Контейнер **легче** VM, потому что делит **ядро хоста**.
- **Dockerfile** собирает образ; порядок инструкций важен из-за **кэша слоёв** — зависимости копируй раньше кода.
- Контейнер **эфемерный** → данные храни в **томах**.
- Порты надо **пробрасывать** (`-p`), наружу сами не торчат.
- **Compose** = много контейнеров одним файлом; сервисы общаются **по имени сервиса**, а не через `localhost`.
