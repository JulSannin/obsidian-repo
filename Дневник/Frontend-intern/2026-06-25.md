## Что я прочитал и освоил

### 1. Что такое React и зачем он нужен

**Проблема чистого JS:** когда UI сложный, ты вручную находишь элементы (`document.querySelector`), меняешь их (`el.textContent = ...`) и следишь, чтобы DOM и данные не разъехались. Это **императивный** подход — ты описываешь, *как* менять.

**React — это библиотека для построения UI.** Подход **декларативный**: ты описываешь, *что* должно быть на экране для текущих данных, а React сам приводит DOM в это состояние.

Идея в одной формуле: **UI = f(state)**. Интерфейс — это функция от состояния. Поменялось состояние → React перерисовал что нужно.

Три кита React:
- **Компоненты** — UI собирается из переиспользуемых кусочков.
- **Состояние (state)** — данные, от которых зависит, что нарисовано.
- **Реактивность** — ты меняешь состояние, а не DOM руками; React обновит DOM сам.

### 2. JSX — что это и как превращается в JS

**JSX** — синтаксис, похожий на HTML, прямо внутри JS. Это **не** строка и не HTML — это синтаксический сахар, который сборщик (Vite/Babel) превращает в вызовы функций.

```jsx
const el = <h1 className="title">Привет, {name}</h1>;
// превращается в:
const el = React.createElement('h1', { className: 'title' }, 'Привет, ', name);
```

Правила JSX, на которых спотыкаются новички:
- **Один корневой элемент** — оборачивай в `<div>` или пустой фрагмент `<>...</>`.
- **`className`** вместо `class`, **`htmlFor`** вместо `for` (т.к. `class`/`for` заняты в JS).
- Атрибуты и стиль — в **camelCase**: `onClick`, `tabIndex`, `style={{ marginTop: 8 }}`.
- **`{...}`** — окно в JS: можно вставить любое **выражение** (не оператор): `{user.name}`, `{2 + 2}`, `{isOk ? 'да' : 'нет'}`.

### 3. Компоненты и пропсы

**Компонент** — функция, которая возвращает JSX. Имя — **с большой буквы** (так React отличает компонент от обычного тега).

```jsx
function Welcome(props) {
  return <h1>Привет, {props.name}</h1>;
}
// использование, как тег:
<Welcome name="Аня" />
```

**Пропсы (props)** — аргументы компонента, данные **сверху вниз** (от родителя к ребёнку). Внутри компонента пропсы **только для чтения** (immutable) — мутировать нельзя.

```jsx
function Card({ title, children }) {   // деструктуризация пропсов
  return (
    <div className="card">
      <h2>{title}</h2>
      {children}             {/* всё, что между тегами компонента */}
    </div>
  );
}

<Card title="Профиль">
  <p>Контент внутри попадёт в children</p>
</Card>
```

**Поток данных однонаправленный (one-way):** данные текут вниз через пропсы. Чтобы ребёнок «сообщил наверх», родитель передаёт ему **функцию-колбэк**.

### 4. Состояние: useState

**Состояние (state)** — данные компонента, которые меняются со временем и при изменении вызывают **ре-рендер** (перерисовку). Обычная переменная не подойдёт — React не узнает, что её надо перерисовать.

```jsx
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0); // [значение, сеттер], начальное 0

  return (
    <button onClick={() => setCount(count + 1)}>
      Кликов: {count}
    </button>
  );
}
```

Ключевые правила:
- **Не меняй состояние напрямую** (`count++`) — только через сеттер `setCount`. Иначе ре-рендера не будет.
- **Обновление асинхронно**: после `setCount` переменная `count` в этом же рендере не изменится — новое значение придёт в следующем рендере.
- Если новое значение зависит от старого — передавай **функцию**: `setCount(c => c + 1)` (безопасно при нескольких обновлениях подряд).
- Состояние-объект/массив меняй **иммутабельно** — создавай новый: `setUser({ ...user, name: 'Аня' })`.

### 5. Рендеринг списков и условный рендеринг

**Список** — через `.map()`, каждый элемент получает **уникальный `key`**:

```jsx
{users.map(u => <li key={u.id}>{u.name}</li>)}
```

> [!warning] key — не для красоты
> `key` нужен React, чтобы понимать, какой элемент изменился/добавился/удалился. Бери **стабильный id**, а не `index` — `key={index}` ломает состояние и порядок при вставке/удалении/сортировке.

**Условный рендеринг** — обычные JS-выражения:

```jsx
{isLoading && <Spinner />}              // && — показать, если true
{user ? <Profile /> : <LoginButton />}  // тернарник — выбор из двух
```

### 6. События и формы (controlled inputs)

События — в camelCase, в `{}` передаём **функцию** (а не вызываем её):

```jsx
<button onClick={handleClick}>OK</button>      // правильно
<button onClick={handleClick()}>OK</button>     // ОШИБКА: вызовется при рендере
```

**Управляемый инпут (controlled)** — значение поля держит **состояние React**, а не сам DOM. Источник правды один — state.

```jsx
function Form() {
  const [name, setName] = useState('');

  return (
    <form onSubmit={e => { e.preventDefault(); console.log(name); }}>
      <input value={name} onChange={e => setName(e.target.value)} />
    </form>
  );
}
```

Связка: `value` ← state, `onChange` → обновляет state. `e.preventDefault()` в `onSubmit` — чтобы страница не перезагружалась.

### 7. Подъём состояния (lifting state up)

Когда **двум соседним компонентам** нужны одни данные — состояние поднимают к их **общему родителю** и передают вниз: значение — пропсом, изменение — колбэком.

```jsx
function Parent() {
  const [value, setValue] = useState('');
  return (
    <>
      <Input value={value} onChange={setValue} />  {/* вниз: данные + колбэк */}
      <Preview value={value} />                    {/* вниз: те же данные */}
    </>
  );
}
```

Правило: **состояние живёт на ближайшем общем предке** тех, кому оно нужно. Данные — вниз, события — вверх.

### 8. useEffect — побочные эффекты

**Побочный эффект** — всё, что выходит за рамки «посчитать JSX из пропсов и состояния»: запрос к API, таймер, подписка, ручная работа с DOM. Для этого — `useEffect`.

```jsx
useEffect(() => {
  // эффект: выполнится после рендера
  const id = setInterval(tick, 1000);

  return () => clearInterval(id); // очистка: перед след. эффектом и при размонтировании
}, [/* зависимости */]);
```

**Массив зависимостей** решает, *когда* эффект перезапускается:
- `[]` — **один раз** после монтирования (и очистка при размонтировании).
- `[a, b]` — после каждого рендера, где изменились `a` или `b`.
- нет массива — после **каждого** рендера (нужно редко).

> [!warning] Главные ловушки useEffect
> - **Не врать в зависимостях**: всё, что используешь внутри, должно быть в массиве (иначе словишь «устаревшее» значение — stale closure).
> - **Очистка обязательна** для таймеров/подписок/запросов — иначе утечки и гонки.
> - **Лишний эффект** — частая ошибка: если значение можно **вычислить при рендере**, эффект не нужен (см. §11 про `useMemo`).

### 9. Запросы к API и старт проекта (Vite)

Загрузка данных — классический эффект. Держим три состояния: **loading / error / data (success)**.

```jsx
function Users() {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let alive = true;                       // защита от гонки
    fetch('/api/users')
      .then(r => { if (!r.ok) throw new Error(`HTTP ${r.status}`); return r.json(); })
      .then(d => alive && setData(d))
      .catch(e => alive && setError(String(e)))
      .finally(() => alive && setLoading(false));
    return () => { alive = false; };        // игнорируем ответ размонтированного компонента
  }, []);

  if (loading) return <p>Загрузка…</p>;
  if (error)   return <p>Ошибка: {error}</p>;
  return <ul>{data.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
```

**Vite** — современный сборщик и дев-сервер, стандарт для старта React-проекта:

```bash
npm create vite@latest my-app -- --template react-ts
cd my-app && npm install && npm run dev
```

(флаг `alive` — это ручной способ; на junior+ эту боль забирает TanStack Query, см. §15.)

### 10. React Router — навигация

В SPA страница **не перезагружается** — роутер подменяет компоненты по URL на клиенте.

```jsx
import { BrowserRouter, Routes, Route, Link } from 'react-router-dom';

<BrowserRouter>
  <nav><Link to="/">Главная</Link> <Link to="/users/42">Профиль</Link></nav>
  <Routes>
    <Route path="/" element={<Home />} />
    <Route path="/users/:id" element={<User />} />   {/* :id — динамический параметр */}
    <Route path="*" element={<NotFound />} />          {/* 404 */}
  </Routes>
</BrowserRouter>
```

`<Link>` вместо `<a>` (чтобы не было перезагрузки), `useParams()` достаёт `:id`, `useNavigate()` — переход из кода.

### 11. Хуки глубже: useRef, useMemo, useCallback, useReducer

Это ядро Junior+. Коротко — зачем каждый:

**`useRef`** — «коробка», которая **переживает ре-рендеры**, но при изменении **НЕ** вызывает перерисовку. Два применения: ссылка на DOM-элемент и хранение мутабельного значения (id таймера, предыдущее значение).

```jsx
const inputRef = useRef(null);
// ...
<input ref={inputRef} />
<button onClick={() => inputRef.current.focus()}>В фокус</button>
```

**`useMemo`** — **кэширует результат вычисления** между рендерами; пересчитывает, только если изменились зависимости. Для тяжёлых вычислений.

```jsx
const sorted = useMemo(() => bigList.sort(cmp), [bigList]); // не сортируем на каждый рендер
```

**`useCallback`** — то же, но **кэширует саму функцию** (чтобы не создавать новую ссылку каждый рендер). Нужен, когда функция уходит в `memo`-компонент или в зависимости другого хука. `useCallback(fn, dep)` ≈ `useMemo(() => fn, dep)`.

**`useReducer`** — альтернатива `useState` для **сложного состояния** с несколькими связанными полями/переходами. Логика — в чистой функции-редьюсере (как в Redux).

```jsx
function reducer(state, action) {
  switch (action.type) {
    case 'inc':   return { count: state.count + 1 };
    case 'reset': return { count: 0 };
    default:      return state;
  }
}
const [state, dispatch] = useReducer(reducer, { count: 0 });
dispatch({ type: 'inc' });
```

### 12. useContext — данные «сквозь» дерево

**Проблема prop drilling:** пропс приходится прокидывать через 5 компонентов, которым он не нужен, ради шестого. **Контекст** даёт «телепорт» — значение доступно любому потомку без передачи через пропсы.

```jsx
const ThemeContext = createContext('light');

function App() {
  return (
    <ThemeContext.Provider value="dark">   {/* раздаём значение поддереву */}
      <Toolbar />
    </ThemeContext.Provider>
  );
}

function Button() {
  const theme = useContext(ThemeContext);  // берём напрямую, без пропсов
  return <button className={theme}>OK</button>;
}
```

Контекст хорош для **редко меняющихся** глобальных вещей: тема, локаль, текущий пользователь. Для часто меняющегося состояния — стейт-менеджер (§15), иначе перерисуется всё поддерево.

### 13. Кастомные хуки

**Кастомный хук** — функция с именем на `use...`, которая **переиспользует логику** на хуках. UI остаётся чистым, логика — отдельно. Это главный способ DRY в React.

```jsx
function useToggle(initial = false) {
  const [on, setOn] = useState(initial);
  const toggle = useCallback(() => setOn(o => !o), []);
  return [on, toggle];
}

// в компоненте:
const [isOpen, toggleOpen] = useToggle();
```

Можно вынести загрузку данных (`useFetch`), debounce (`useDebounce`), отслеживание размера окна (`useWindowSize`) — всё, что повторяется.

### 14. Оптимизация ре-рендеров

Компонент перерисовывается, когда: меняется его **state**, меняются **пропсы** или перерисовался **родитель**. Лишние ре-рендеры — частая причина тормозов.

Инструменты (применять **по факту проблемы**, не везде подряд):
- **`React.memo(Component)`** — пропускает ре-рендер, если пропсы не изменились (поверхностно).
- **`useMemo`** — не пересоздавать тяжёлые значения / объекты-пропсы.
- **`useCallback`** — не пересоздавать функции-пропсы (иначе `memo` бесполезен — новая ссылка каждый раз).

```jsx
const Child = React.memo(function Child({ onClick }) { /* ... */ });

function Parent() {
  const handle = useCallback(() => {/* ... */}, []); // стабильная ссылка → memo сработает
  return <Child onClick={handle} />;
}
```

> [!tip] Не оптимизируй вслепую
> `useMemo`/`useCallback` сами стоят памяти и усложняют код. Сначала измерь (React DevTools → Profiler), потом оптимизируй узкое место.

### 15. Управление состоянием: серверное vs клиентское

На junior+ важно делить состояние на два вида — у них **разные инструменты**:

| Вид | Что это | Чем управлять |
| --- | --- | --- |
| **Серверное** | данные с бэкенда (список юзеров, товары) | **TanStack Query** (React Query) |
| **Клиентское** | UI-состояние (тема, корзина, модалки) | **Zustand** / **Redux Toolkit** |

**TanStack Query** убирает ручную возню с `loading/error` и `useEffect`-загрузкой: даёт **кэш**, авто-**рефетч**, дедупликацию и инвалидацию.

```jsx
const { data, isLoading, error } = useQuery({
  queryKey: ['users'],
  queryFn: () => fetch('/api/users').then(r => r.json()),
});
```

**Zustand** — минималистичный клиентский стор без бойлерплейта:

```jsx
const useStore = create(set => ({
  count: 0,
  inc: () => set(s => ({ count: s.count + 1 })),
}));
const count = useStore(s => s.count); // подписка только на нужный кусок
```

Правило джуна+: **не пихай серверные данные в Redux/Zustand**. Сервер → Query, клиент → Zustand/RTK. Контекст — для редко меняющегося (тема/локаль/auth).

### 16. Правила хуков (Rules of Hooks)

Хуки (`useState`, `useEffect`, …) работают только при двух условиях:
1. **Вызывать только на верхнем уровне** компонента или кастомного хука — **НЕ** внутри условий, циклов, вложенных функций. React связывает хуки с компонентом **по порядку вызова**; условие сдвинет порядок → всё сломается.
2. **Вызывать только из React-функций** — компонентов или других хуков (не из обычных функций).

```jsx
// ❌ нельзя
if (user) { const [x, setX] = useState(0); }

// ✅ можно
const [x, setX] = useState(0);
if (user) { /* используем x */ }
```

ESLint-плагин `eslint-plugin-react-hooks` ловит нарушения и подсказывает зависимости эффектов.

### 17. Что запомнить

- **UI = f(state)**: описываешь, *что* нарисовать для данных, React сам обновляет DOM.
- **JSX** — сахар над `createElement`; `className`, camelCase, один корень, `{выражение}`.
- **Пропсы — вниз и только для чтения**, события — вверх через колбэки (one-way data flow).
- **Состояние — через сеттер и иммутабельно**; обновление асинхронно; новое от старого — через функцию.
- **`key` — стабильный id**, не index.
- **Управляемые инпуты**: источник правды — это state (`value` + `onChange`).
- **`useEffect`** — для побочных эффектов; честный массив зависимостей + очистка; лишний эффект — частая ошибка.
- Junior+ хуки: `useRef` (без ре-рендера), `useMemo`/`useCallback` (кэш значения/функции), `useReducer` (сложное состояние), `useContext` (без prop drilling).
- **Кастомные хуки** — переиспользование логики (DRY).
- **Оптимизация (`memo`/`useMemo`/`useCallback`) — по факту проблемы**, не вслепую.
- **Серверное состояние → TanStack Query**, клиентское → Zustand/Redux Toolkit. Не смешивай.
- **Правила хуков**: только на верхнем уровне и только из React-функций.
